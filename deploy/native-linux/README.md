# Obsideo Data Farmer - Ubuntu/Debian

Run an Obsideo storage provider on an Ubuntu or Debian server and earn AKT.

> **Want to know exactly what you're running?** See
> [TRANSPARENCY.md](../../TRANSPARENCY.md) for a complete inventory of source
> files, network calls, storage behavior, and dependencies. See
> [SECURITY.md](../../SECURITY.md) for the encryption and trust model.

## Requirements

- Ubuntu 22.04+ or Debian 12+ with systemd
- A Tailscale account (free) with Funnel enabled in your tailnet ACL
- Public internet access (Tailscale Funnel gives you a free HTTPS endpoint)
- At least 10 GB of free disk space for storage

## Quick start

```bash
unzip obsideo-provider-ubuntu.zip
cd obsideo-provider-ubuntu
sudo bash setup.sh
```

The script installs Tailscale if missing, creates a service user, installs the
binary to `/opt/obsideo-provider`, writes a systemd unit, brings the provider
online, and registers it with the coordinator.

It is **safe to re-run** at any time. Existing identity, provider ID, and
storage are preserved.

## Before you start

1. **Create a Tailscale account** at https://tailscale.com (free).
2. **Enable Funnel in your ACL policy**. In the Tailscale admin console,
   open Access Controls and add:
   ```json
   "nodeAttrs": [
     {
       "target": ["autogroup:member"],
       "attr": ["funnel"]
     }
   ]
   ```
3. **Authenticate Tailscale** before running setup:
   ```bash
   sudo tailscale up
   ```
   Follow the auth URL it prints.

   Or for unattended installs, pass an auth key directly to setup:
   ```bash
   sudo TS_AUTHKEY=tskey-auth-XXX bash setup.sh
   ```

## What setup.sh does

1. Installs dependencies: `curl`, `jq`, `openssl`, `ca-certificates`
2. Installs Tailscale (if missing) and checks it is authenticated
3. Creates the `obsideo` service user and `/opt/obsideo-provider` layout
4. Installs the `datafarmer` binary
5. Generates an ed25519 identity keypair (first run only)
6. Fetches the coordinator's public key
7. Configures Tailscale Funnel to expose `localhost:3334` publicly
8. Writes `config.yaml` and `/etc/obsideo-provider/provider.env`
9. Installs and starts the `obsideo-provider` systemd service
10. Waits for the provider to respond on `localhost:3334/health`
11. Registers with the coordinator (retries up to 10 times)
12. Restarts the service with the assigned `provider_id`

## After setup

Your provider starts in `pending` status. The coordinator admin must approve it
before it receives uploads. The setup output shows the provider ID.

## Operating the provider

```bash
# Check status
systemctl status obsideo-provider

# Follow logs
journalctl -u obsideo-provider -f

# Restart
sudo systemctl restart obsideo-provider

# Check farm earnings
sudo -u obsideo /opt/obsideo-provider/datafarmer info \
    --config /opt/obsideo-provider/config.yaml
```

## Changing configuration

Edit `/etc/obsideo-provider/provider.env` to change the coordinator URL,
capacity, or wallet, then re-run setup:

```bash
sudo bash setup.sh
```

Re-running reads the current values from `provider.env` and skips the
interactive prompts.

## Using custom storage (ZFS mount, external drive)

Point `OBSIDEO_DATA_DIR` at the mount before running setup:

```bash
sudo OBSIDEO_DATA_DIR=/mnt/obsideo-pool bash setup.sh
```

The service user needs write access to that directory. Setup handles this
automatically on first run.

## Files installed

```
/opt/obsideo-provider/
  datafarmer               # the binary
  config.yaml              # generated by setup
  coordinator_pub.pem      # fetched from coordinator
  data/                    # object storage + identity key
    .identity_ed25519
    objects/

/etc/obsideo-provider/
  provider.env             # operator-editable config
  state.json               # provider_id (do not delete)

/etc/systemd/system/
  obsideo-provider.service
```

## Uninstalling

```bash
# Stop the service, remove binary + systemd unit, keep data:
sudo bash uninstall.sh

# Stop the service and wipe everything including data and user:
sudo bash uninstall.sh --purge
```

## Troubleshooting

**"Tailscale is not authenticated"**
Run `sudo tailscale up` first. Then re-run setup.

**"tailscale serve failed - is Funnel enabled"**
You need to enable the `funnel` nodeAttr in your Tailscale ACL. See
https://tailscale.com/kb/1223/funnel

**"registration failed after 10 attempts"**
The coordinator did a liveness check on your Tailscale Funnel URL and couldn't
reach it. Common causes:
- Funnel not enabled in ACL (most common)
- Funnel still propagating (wait 30s and re-run setup)
- Provider is serving but the public URL isn't routing (check `curl https://$(tailscale status --json | jq -r '.Self.DNSName' | sed 's/\.$//')/health` from an external box)

**"provider did not respond on localhost:3334/health within 60s"**
Check journalctl for binary errors:
```bash
journalctl -u obsideo-provider -n 50
```

Common causes:
- Data directory not writable by `obsideo` user
- Port 3334 already in use by another service
- Binary failed to load config

**Provider shows as 'pending' forever**
The coordinator admin needs to approve it manually. Contact them with the
provider ID shown at the end of setup.
